# 同方泰德水印替换工具 v3.7

自动将维保照片中的旧水印替换为**同方泰德**水印（logo + 标题 + 维保单位）。

## 文件说明

```
watermark_replacer.py        # 主程序
header_tongfangtaider.jpg    # 顶部素材图（仅用于裁出左侧 logo）
bar_tongfangtaider.jpg       # 旧版底部素材图（v3.3 起不再使用，保留仅为兼容）
config.txt                   # 默认标题/维保单位配置（首次运行会自动生成）
```

## 安装依赖

```bash
pip install -r requirements.txt
```

## 最简单的用法（推荐，适合日常批量处理）

1. 双击运行一次 `watermark_replacer.exe`（或不带参数运行脚本），
   程序会在同目录下生成一个 `config.txt` 模板。
2. 用记事本打开 `config.txt`，把 `site=` 改成这批照片的标题，
   `unit=` 改成维保单位（一般固定不用改），保存。
3. 以后只要把图片（或整个文件夹里的图片）**直接拖到 exe 图标上**，
   就会自动用 `config.txt` 里的内容处理，输出到同目录的 `output` 文件夹。

`config.txt` 示例：

```
site=中国农业银行北方数据中心
unit=同方泰德国际科技（北京）有限公司
```

## 命令行用法（适合一次性处理 / 跑不同标题）

```bash
# 单张，命令行参数会覆盖 config.txt 里的对应项
python watermark_replacer.py photo.jpg --site "中国农业银行北方数据中心" --unit "同方泰德国际科技（北京）有限公司"

# 批量
python watermark_replacer.py *.jpg --site "..." --unit "..." -o ./my_output

# 手动指定中文字体（自动查找失败时使用）
python watermark_replacer.py photo.jpg --site "..." --unit "..." --font "C:\Windows\Fonts\msyh.ttc"
```

`--unit` / config.txt 里的 `unit=` 都不需要自己加"维保单位："前缀，程序会自动加上。

## 更新说明

### v3.7：修复左边界被孤立噪声列误判"凸出来"的问题

v3.6 处理了"背景物体被误判进卡片范围"的情况，这次又发现了边界检测代
码本身的一个结构性漏洞：左右边界原本是直接取所有满足亮度条件的列里最
左和最右的那一个，完全没检查这些列彼此是否连续。如果照片最左侧边缘正
好有一列因为渐晕、反光等原因偶然也满足"够亮"的条件——哪怕紧挨着的几
十像素完全不满足，跟真正的卡片之间隔着一段明显的暗区——这一个孤立的
噪声列也会被直接当成卡片边界，导致新水印的边比卡片实际宽度多出一截。

v3.7 改为：列方向的边界检测也用跟"判断卡片整体高度"一样的连续区段
（run）+合并逻辑，只取其中最长的连续段作为左右边界，不再被孤立的噪声
列误导。

### v3.6：修复蓝色条右边缘被背景误判"凸出来"的问题

v3.5 修复了标题条高度忽长忽短的问题后，又发现宽度也会有类似情况：判断
卡片左右边界用的是"亮、纯色"这种宽泛的亮度条件，但照片背景里有些物体
（人物衣领、肤色、白色机柜门等）刚好也满足这个条件，而且恰好紧贴在卡
片右侧、处于跟标题条/维保单位条相同的高度，于是被错误并入了卡片范围，
导致新水印的蓝色条右边缘比卡片实际宽度宽出一截，跟中间没被替换的灰色
信息行右边缘对不齐，看起来像"凸出来了"。

v3.6 改为：标题条和维保单位条各自的右边界，单独在它们所在的精确高度
范围内用蓝色通道差异重新扫描确定，不再依赖容易被浅色背景物体误判的
亮度判断。

### v3.5：修复蓝色标题条检测异常（高度忽长忽短）的问题

`find_card_and_regions()` 里检测顶部蓝色标题条的逻辑有两个边界情况没处理好：

1. **检测过长**：为了把 logo 白底框也框进去，代码会从蓝色条顶部往上逐行
   探测"是否够亮"，但没有给这个探测设上限。如果蓝色条上方刚好是浅色的
   背景（比如水泥地、白色墙面/立柱），探测会一直往上走，把背景误判成
   水印的一部分，导致新水印的蓝色标题条比原图异常变长，往上侵入了照片
   内容。
2. **检测过短**：判断"是不是蓝色"是逐行统计蓝色像素占比，但当标题文字
   笔画比较密集时，某一行的蓝色占比可能短暂跌破阈值，把本应连续的一整
   条蓝色区域错误切成了好几段，而代码只取了第一段，导致检测出的框比实
   际标题矮一截，新文字超出框外显示不全。

v3.5 移除了那段没有上限的"向上探测"逻辑（蓝色条顶部本身已经测得足够准
确，不需要再额外扩展），并在切分蓝色区域后加了一步小间隙合并：只要两
段蓝色区域之间的间隔很小（远小于标题区和维保单位区之间的正常间隔），
就认为它们本来是同一条，自动合并。修复后在两张不同风格的样本图上测得
的标题条高度一致（约 400px），不再出现忽长忽短的情况。

### v3.4：支持 config.txt，修复拖拽/双击无法使用的问题

v3.3 把 `--site`/`--unit` 设成了命令行必填参数，结果"把图片拖到 exe
图标上"这种最常用的用法直接报错退出（拖拽时没办法附加命令行参数）。

v3.4 新增 `config.txt` 配置文件作为默认值来源，双击或拖拽时自动读取；
命令行传 `--site`/`--unit` 时优先用命令行的值。首次运行如果还没配置，
会自动生成模板并提示去填写，而不是直接报错。

### v3.3：修复文字挤压变形的问题

v3.2 是把"预先画好文字的固定图片"强制拉伸塞进检测到的水印框里。当新单位
名称比旧的长、需要换行时，预渲染图和实际检测框的高度不匹配，文字会被
挤压变形（看起来像边框装不下内容）。

v3.3 改为：

1. 只从 `header_tongfangtaider.jpg` 裁出左侧 logo 图标，按比例缩放（不拉伸变形）。
2. 标题、维保单位文字实时绘制，自动从大字号往小尝试，优先单行显示；
   单行实在放不下才换两行，并重新计算字号让两行都能完整放进框内。

## 中文字体

程序运行时需要系统里有中文字体来绘制文字，会按顺序自动查找：

```
C:\Windows\Fonts\msyhbd.ttc   微软雅黑 Bold
C:\Windows\Fonts\msyh.ttc     微软雅黑
C:\Windows\Fonts\simhei.ttf   黑体
```

正常的中文 Windows 系统都自带以上字体，无需额外配置。如果找不到，
程序会报错提示，可用 `--font "字体文件路径"` 手动指定。

字体查找逻辑只依赖系统标准路径，与是否打包成 exe 无关，打包后同样有效。

## 检测原理

通过识别照片中的半透明白色/蓝色水印卡片精确定位水印区域，再在卡片
内部分别找顶部蓝色标题区和底部蓝色维保单位栏的位置和尺寸。

## EXE 生成

上传所有文件后，点击仓库 **Actions** 标签，等待自动打包完成，
在 Artifacts 中下载 `watermark_replacer_exe.zip`。

打包出的 exe 用法和上面一致：双击运行一次生成/编辑 `config.txt`，
之后把图片拖到 exe 图标上即可；也可以在命令行用 `--site`/`--unit` 覆盖。
